1. PURPOSE
   - Manages file organization rules used by FormPreviewOrganize
   - Supports two mutually exclusive modes:
     * Batch-generated rules (via 5 batch processors)
     * Custom user-defined rules (max 10)
   - Reads/writes the organization_rules database table

2. FIELDS & PROPERTIES
   - private string baseDestinationPath = "";
     Reads from app_settings on form load; used to auto-fill rule destinations
     and display the fallback folder path.

3. CONSTRUCTOR
   - public FormConfigureRules()
   - Calls InitializeComponent() then SetInitialCheckboxStates()

5. KEY METHODS

   private void SetInitialCheckboxStates() (line 17)
   - Disables all condition controls (cmbExtCategory, cmbAgeGroup, txtKeyword,
     nudSizeMin, nudSizeMax)

   private int CountRules(string whereClause) (line 26)
   - Returns COUNT(*) from organization_rules with optional WHERE clause
   - Used to determine batch vs custom rule counts

   private void UpdateFormState() (line 48)
   - Core state management method
   - If total rules == 0: everything is enabled (batch + custom)
   - If batch rules exist: batch buttons disabled, custom form locked,
     shows "Batch rules exist. Custom rules are locked."
   - If only custom rules exist (batch == 0):
     * If custom >= 10: locked, shows "Maximum 10 custom rules reached"
     * If custom < 10: enabled, shows "Custom rules mode (N/10 used)"
   - All batch buttons (btnBatchType/Age/Alphabet/Size/Date) are disabled
     whenever ANY rules exist

   private void LoadRulesFromDatabase() (line 170)
   - Queries organization_rules WHERE is_active = 1
   - Displays in listViewRules with columns: ID, Name, Conditions, Destination,
     Source
   - Conditions column built by BuildConditionsString()

   private string BuildConditionsString(MySqlDataReader reader) (line 212)
   - Builds a human-readable conditions string from DB fields:
     Ext: ..., Age: ..., Key: ..., Size: ...
   - Size formatted with FormatSize()
   - Returns empty if no conditions

   private void ResetFormFields() (line 369)
   - Clears all custom rule form fields to default state

   private void InsertBatchRule(MySqlConnection conn, ...) (line 677)
   - Inserts a single rule with rule_source = 'batch'
   - Uses the same INSERT statement as custom rules but with fixed source

   private string FormatSize(long bytes) (line 697)
   - Human-readable byte formatting (B, KB, MB, GB)

8. MODE STATE MACHINE
   ┌─────────────┐   rules=0   ┌──────────────┐
   │  No Rules   │ ──────────> │ Everything   │
   │ (fresh)     │ <────────── │ Enabled      │
   └─────────────┘   delete all └──────────────┘
                        │
                        │ user creates batch or custom
                        v
   ┌──────────────────────────────────┐
   │ Rules Exist (batch or custom)    │
   │ - Batch buttons always disabled  │
   │ - If batch>0: custom locked      │
   │ - If batch=0, custom<10: custom  │
   │   form enabled                   │
   │ - If batch=0, custom>=10: locked │
   └──────────────────────────────────┘

9. LOGIC FLOW SUMMARY
   Load -> read base destination path -> LoadRulesFromDatabase() ->
   UpdateFormState()
   On save/batch/delete -> reload rules list -> update form state
   Batch processors are mutually exclusive with all custom rules; you must
   delete all rules to switch modes
